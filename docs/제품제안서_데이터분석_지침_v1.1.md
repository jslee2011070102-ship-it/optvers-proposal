# 제품 제안서 전용 데이터 분석 지침 (v1.1)

> 최초 작성: 2026년 3월
> 적용 범위: 쿠팡 시장 데이터 기반 신제품 제안서 전 과정
> v1.1 업데이트: 검증 에이전트 시스템 추가 (2026년 3월)

---

## 제0원칙: 절대적 사실성 (Principle of Absolute Factual Integrity)

모든 분석의 최상위 원칙이며, 다른 모든 규칙에 우선한다.

**환각(Hallucination) 절대 금지**
업로드된 데이터 파일에 명시적으로 존재하지 않는 수치, 순위, 상품명, 비율을 추측하거나 창작하지 않는다.
예) "고체탈취제 상품이 2개뿐"처럼 데이터에 없는 숫자를 추정으로 채우는 행위 → 분석 실패로 간주.

**'수치'와 '판단'의 엄격한 분리**
모든 분석은 두 단계로 명확히 분리되어야 한다.
- **수치** = 데이터 파일에서 직접 읽힌 것
- **판단** = 수치로부터 도출한 해석

판단은 반드시 100% 검증된 수치에만 근거할 수 있다.

**불확실성 명시 의무**
데이터에 없거나 추가 확인이 필요한 경우, 추정으로 메우는 대신 아래 표기 중 하나를 반드시 사용한다.

| 상황 | 표기 방식 |
|---|---|
| 데이터 파일에 해당 수치 없음 | `[데이터 없음 — 직접 확인 필요]` |
| 수치는 있으나 해석이 불확실함 | `[추정 — 근거: OOO]` |
| 데이터 범위 밖의 주장 | `[데이터 범위 초과 — 해당 파일로 검증 불가]` |

**임무 실패 조건**
단 하나의 수치라도 데이터 출처 없이 사용되거나, 데이터와 다르게 왜곡된 경우 해당 섹션 전체를 재작성한다.

## 단계 0: 데이터 파일 목록 선언 (Data Source Declaration)

분석 시작 전, 사용 가능한 데이터 파일 전체를 아래 형식으로 먼저 선언한다.
이후 모든 수치는 반드시 이 목록 중 하나를 출처로 명시해야 한다.

```
[파일A] 키워드 분석 파일
  포함 데이터: 월 검색량, 작년 총 검색량, 월간 총 판매량, 월간 총 매출,
               배송 비율, 리뷰 분포, 포화도, 인기키워드(검색량), 자동완성키워드, 상품명 빈도수

[파일B] 카테고리 분석 파일
  포함 데이터: 카테고리 월 총 매출, 카테고리 월 총 판매량,
               배송 비율, 포화도, 리뷰 분포

[파일C] 상품 리스트 파일 (파일A 또는 파일B의 shoppingList 시트)
  포함 데이터: 상품명, 순위, 가격, 리뷰 수, 클릭 수,
               월 판매량, 월 매출, 전환율, 배송방법
```

선언되지 않은 파일의 데이터는 존재하지 않는 것으로 간주한다.

## 단계 1: 원자 단위 데이터 추출 (Raw Data Extraction)

분석 시작과 동시에 업로드된 모든 파일을 컬럼 단위로 분해하여 아래 4개 카테고리로 분류한다.
이 단계에서는 어떤 해석도 가미하지 않으며 순수 수치와 텍스트만 기록한다.

### ① 수요 데이터
월 검색량, 작년 총 검색량, 인기키워드별 검색량.
→ 시장이 얼마나 크고 안정적인지를 보여주는 원재료.

### ② 경쟁 데이터
포화도(매출/판매/리뷰), 배송 방식 비율, 상위 상품 전환율, 리뷰 수 분포.
→ 시장 내 경쟁 강도를 보여주는 원재료.

### ③ 공백 데이터
인기키워드 중 상품명 미포함 키워드, 자동완성 패턴, 상위 상품 공통 속성 분석.
→ 진입 가능한 틈새를 보여주는 원재료.

### ④ 수익 데이터
상위 상품 가격대 분포, 월 매출, 월 판매량, 전환율.
→ 실제로 돈이 되는지를 보여주는 원재료.

**출처 표기 규칙**
모든 수치 뒤에는 반드시 `[파일명 · 컬럼명]` 형식으로 출처를 표기한다.

```
예) 월 검색량 3,160회 [파일A · 월 검색량]
    1~3위 매출 포화도 43.1% [파일A · 1-3위 매출 포화도]
    카테고리 월 매출 59.8억 [파일B · 월간 총 매출]
```

## 단계 2: 3단계 해석 프로세스 (Interpretation Process)

단계 1에서 추출된 수치만을 근거로 아래 3단계 순서로 해석을 진행한다.
각 단계의 판단 기준은 사전에 정의된 기준값과 대조하며, 기준값이 없는 항목은 판단 보류로 처리한다.

---

### Step 1 — 시장 확인: "돈이 되는 시장인가?"

| 항목 | 출처 | ◎ 기준 | △ 기준 | ✕ 기준 |
|---|---|---|---|---|
| 월 검색량 | 파일A · 월 검색량 | 5,000회+ | 1,000~4,999회 | 999회- |
| 검색 안정성 | 파일A · 작년 총 검색량 | 총검색량÷12 ≈ 월검색량 (편차 2배 미만) | 편차 2~3배 (계절성 주의) | 편차 3배 이상 |
| 카테고리 월 매출 | 파일B · 월간 총 매출 | 30억+ | 10~29억 | 10억- |
| 카테고리 월 판매량 | 파일B · 월간 총 판매량 | 5만개+ | 1만~4.9만개 | 1만개- |

**Step 1 통과 기준**: 4개 항목 중 3개 이상 ◎이면 통과. 2개 이하면 "시장 재검토" 판정.

---

### Step 2 — 경쟁 구조: "들어갈 틈이 있는가?"

| 항목 | 출처 | ◎ 기준 | △ 기준 | ✕ 기준 |
|---|---|---|---|---|
| 1위 매출 포화도 | 파일A · 1위 매출 포화도 | 20% 미만 | 20~35% | 35% 초과 |
| 1~3위 매출 포화도 | 파일A · 1-3위 매출 포화도 | 50% 미만 | 50~70% | 70% 초과 |
| 1~3위 리뷰 포화도 | 파일A · 1-3위 리뷰 포화도 | 40% 미만 | 40~60% | 60% 초과 |
| 로켓 계열 배송 비율 | 파일A · 로켓+판매자로켓 비율 합산 | 70% 미만 | — | 80% 이상 (로켓 등록 필수) |
| 상위 5개 평균 전환율 | 파일C · 전환율 (상위 5개 평균) | 10%+ | 5~9% | 5% 미만 |

**Step 2 통과 기준**: 매출 포화도 2개 항목(1위/1~3위)이 모두 ◎이면 통과.
단, 리뷰 포화도 △·✕, 로켓 비율 ✕ 항목은 "진입 전제 조건"으로 별도 명시.

---

### Step 3 — 빈자리 발견: "제품 방향을 특정할 수 있는가?"

이 단계는 수치 대입이 아닌 패턴 분석이므로, 각 근거를 데이터에서 인용한 원문과 함께 기록한다.

| 항목 | 출처 | 분석 방법 | 빈자리 판단 기준 |
|---|---|---|---|
| 고검색/저공급 키워드 | 파일A · 인기키워드 + 파일C · 상품명 | 인기키워드 상위 5개 → 파일C 상품명에 해당 키워드 포함 상품 수 **직접 카운트** | 검색량 5,000+ 인데 상품명 포함 3개 이하 |
| 자동완성 패턴 분류 | 파일A · 자동완성키워드 | 형태 / 성분 / 용도 / 향기 로 분류 | 특정 분류에 2개 이상 등장 |
| 상위 상품 공통 약점 | 파일C · 상품명 | 상위 5개 상품명에서 공통적으로 없는 속성 추출 | 3개 이상 상품에서 동일 속성 부재 |
| 가격 공백대 | 파일C · 가격 | 5,000원 구간으로 분포 정리 | 상품 수 0~1개인 구간 |

> **⚠️ 중요**: "고검색/저공급 키워드"의 상품 수 카운트는 반드시 파일C 상품명을 직접 카운트한 결과만 사용한다. 추정값 사용 절대 금지.

**Step 3 통과 기준**: 고검색/저공급 키워드 1개 이상 + 자동완성 패턴 2개 이상 교차 확인 시 "제품 방향 특정 가능" 판정.

---

## 단계 3: 최종 판정 및 전제 조건 명시 (Final Verdict)

3개 Step 결과를 종합하여 아래 4가지 판정 중 하나를 선택한다.
판정문에는 반드시 각 Step의 결과 요약과 데이터 출처가 함께 기재되어야 한다.

| 판정 | 조건 |
|---|---|
| **적극 추천** | Step 1 통과 + Step 2 통과(전제 조건 없음) + Step 3 빈자리 명확 + 검증 에이전트 통과 |
| **조건부 추천** | Step 1 통과 + Step 2 통과(단, 전제 조건 존재) + Step 3 빈자리 1개 이상 + 검증 에이전트 통과 |
| **틈새 제안** | Step 1 통과 + Step 2 포화도 △ + Step 3 롱테일 키워드 공백 발견 + 검증 에이전트 통과 |
| **재검토** | Step 1 미통과 또는 Step 2 포화도 ✕ 또는 Step 3 빈자리 없음 또는 검증 에이전트 불합격 |

**전제 조건 명시 규칙**
"조건부 추천" 판정 시, Step 2에서 △ 또는 ✕로 판정된 항목은 반드시
"이 조건을 충족해야 진입 가능하다"는 문장으로 별도 명시한다.
전제 조건 없이 추천만 쓰는 것은 금지.

## 단계 4: 제품 방향 특정 (Product Direction)

Step 3에서 확인된 빈자리를 근거로 제품 방향을 특정한다.
모든 항목의 근거는 반드시 데이터에서 인용하며, 데이터에 없는 항목은 `[데이터 없음]`으로 표기한다.

```
제품명 방향:  [OOO]
근거 키워드:  [파일A 인기키워드에서 인용] — OOO건 검색, 상품명 포함 OO개 확인 [파일C]
성분/형태:    [자동완성 패턴에서 도출] — 파일A 자동완성키워드 "OOO" 등장
가격 포지션:  [파일C 가격 분포에서 공백 확인] — OOO~OOO원 구간 상품 O개
상품명 예시:  [근거 키워드 조합]
제조 방식:    [데이터 없음 — 고객사 확인 필요]
```

> **⚠️ 주의**: 제조 방식, 원가, ODM 업체 정보는 데이터 파일에 존재하지 않으므로
> 반드시 `[데이터 없음]` 표기 후 고객사 확인 사항으로 처리한다.
> 임의로 "국내 ODM 활용" 등을 작성하는 것은 금지.

## 단계 5: 예상 매출 시나리오 (Revenue Scenario)

매출 시나리오는 데이터 기반 수치와 가정값을 반드시 구분하여 표기한다.

### 데이터 기반 수치 (파일에서 직접 추출)
- 카테고리 월 총 매출 `[파일B · 월간 총 매출]`
- 카테고리 월 총 판매량 `[파일B · 월간 총 판매량]`
- 상위 5개 평균 전환율 `[파일C · 전환율]`
- 상위 상품 가격대 분포 `[파일C · 가격]`

### 가정값 (데이터 없음 — 반드시 명시)
- 신제품 목표 점유율: `[가정값 — 고객사와 협의 필요]`
- 판매대행 수수료율: `[가정값 — 계약 조건에 따라 상이, 직접 입력 필요]`
- 초기 리뷰 확보 비용: `[가정값 — 마케팅 예산에 따라 상이]`

### 시나리오 작성 규칙

보수·중간·낙관 3가지 시나리오를 작성하되,
각 시나리오의 판매량 가정 근거를 반드시 명시한다.

```
보수적: 카테고리 월 판매량 [파일B]의 OO% 점유 가정 → OOO개
중간:   카테고리 월 판매량 [파일B]의 OO% 점유 가정 → OOO개
낙관적: 카테고리 월 판매량 [파일B]의 OO% 점유 가정 → OOO개

수수료: [가정값 OO% — 실제 계약 조건으로 교체 필요]
```

## 단계 6: 이중 검증 시스템 (Dual Validation System)

제안서 제출 전 두 단계의 검증을 반드시 통과해야 한다.

---

### 6-1단계: 자기검증 (Self-Check)

작성자가 수행하는 데이터 무결성 검증. 아래 체크리스트를 반드시 통과해야 한다.
하나라도 ✕이면 해당 섹션을 재작성한다.

```
✅ 검증 1 — 출처 없는 수치 스캔
   모든 수치에 [파일명 · 컬럼명] 출처가 표기되어 있는가?
   데이터 파일에 없는 수치가 출처 없이 사용되지 않았는가?

✅ 검증 2 — 수치/판단 분리 확인
   "데이터에서 읽힌 것"과 "도출한 판단"이 문장 안에서 혼재되지 않는가?
   판단 문장 앞에 근거 수치가 먼저 제시되어 있는가?

✅ 검증 3 — 불확실성 표기 확인
   데이터에 없는 항목(제조방식, 수수료율 등)에 [데이터 없음] 또는 [가정값] 표기가 되어 있는가?
   추정으로 채워진 빈칸이 없는가?

✅ 검증 4 — Step 3 상품 카운트 재확인
   고검색/저공급 키워드 분석의 상품 수가 파일C 실제 상품명을 직접 카운트한 결과인가?
   카운트 결과와 제안서 본문의 수치가 일치하는가?

✅ 검증 5 — 판정 근거 완결성 확인
   최종 판정에 Step 1~3 결과 요약이 모두 포함되어 있는가?
   "조건부 추천" 판정 시 전제 조건이 명시되어 있는가?
```

---

### 6-2단계: 검증 에이전트 (Validation Agent) ★ 권장

**목적**: 작성자의 자기합리화를 차단하기 위한 독립적 검증

제안 문서 작성 완료 후, **독립적인 검증 에이전트**가 다음을 평가한다:

#### 검증 에이전트가 평가하는 항목

1. **카테고리 적합성**
   - 제안된 제품이 해당 카테고리에서 실제로 팔리거나 팔릴 수 있는가?
   - 카테고리와 무관한 제품이 제안되지 않았는가?
   - 예시 불합격: 성인용품 카테고리에서 "페이스 마스크" 제안

2. **실제 구매 가능성**
   - 이 카테고리의 고객이 정말 이 제품을 사겠는가?
   - 단순 논리적 가능성이 아닌 **현실의 구매 행동** 기준
   - "데이터에는 검색량이 있지만 실제 구매자가 있는가?"

3. **제품명 논리 검증**
   - 제품명이 논리적으로 성립하는가?
   - 서로 다른 제품 카테고리 키워드가 결합되지 않았는가?
   - 브랜드명이 포함되지 않았는가?
   - 마케팅 문구가 제품명으로 사용되지 않았는가?
   - 예시 불합격: "마스크+바이브레이터", "코멧 포함", "쉽게 사용"

4. **키워드 카테고리 분류 검증**
   - 각 제안 제품이 단일 카테고리 키워드에서만 도출되었는가?
   - 서로 다른 제품 카테고리의 키워드를 강제로 결합하지 않았는가?

#### 검증 에이전트의 판정 규칙

```
✅ 합격 (PASS)
   - 모든 제안 제품이 카테고리에 적합
   - 고객의 실제 구매 가능성이 높음
   - 제품명이 논리적으로 성립
   - 제품 카테고리가 혼용되지 않음

❌ 불합격 (FAIL)
   - 카테고리 외부 제품 제안
   - 제품명에 브랜드명 포함
   - 서로 다른 제품 카테고리 키워드 결합
   - 논리적으로 성립하지 않는 제품명
```

#### 불합격 시 조치

불합격 판정 시 **자동 재생성** 프로세스 실행:
1. 검증 에이전트의 불합격 이유 분석
2. 더 엄격한 제약 조건으로 제품 재제안
3. 재검증 (최대 2회 재시도)
4. 2회 재시도 후에도 불합격 시 → 수동 개입 필요

#### 구현 흐름

```
생성 단계:
[Claude] 슬라이드 5, 6, 7 생성
         ↓
검증 단계:
[검증 에이전트] 슬라이드 7의 제품 제안 평가
         ↓
         ├─ ✅ PASS → 최종 확정
         └─ ❌ FAIL → 재생성 (최대 2회)
              ├─ 1회 재시도
              ├─ 2회 재시도
              └─ 여전히 실패 → 수동 확인 필요

최종 출력:
- validationPassed: true/false
- validationAttempts: 1/2/3 (검증 시도 횟수)
```

#### 예시: 성인용품 카테고리

**불합격 예시**:
- 제안: "실리콘 페이스 마스크 대용량 일회용"
- 이유: 카테고리 외부 제품. 성인용품 고객이 구매 불가능

**합격 예시**:
- 제안: "실리콘 바이브레이터 10단계 진동 방수형"
- 이유: 카테고리 적합, 고객 수요 명확, 제품명 논리적

---

**권장 사항**: 모든 제안 문서는 **검증 에이전트 통과 여부**를 최종 판정의 필수 요소로 간주한다.
검증 에이전트 통과 없이는 "최종 추천" 판정 불가.

## 부록: 오류 유형 라이브러리 (Error Pattern Library)

분석을 반복할수록 이 목록에 추가한다.

| 오류 유형 | 발생 사례 | 올바른 처리 |
|---|---|---|
| 수치 환각 | "상품이 2개뿐" → 파일에 없는 카운트를 추정으로 작성 | 파일C 상품명 직접 카운트 후 기재 |
| 가정값 무표기 | "판매대행 수수료 15%" → 협의된 적 없는 수치 사용 | `[가정값 OO% — 계약 조건으로 교체 필요]` |
| 판단의 수치화 | "경쟁이 치열하다" → 근거 수치 없이 판단만 서술 | "1~3위 포화도 43.1% [파일A], 기준치 50% 이하 → 진입 가능 판단" |
| 데이터 범위 초과 | "소비자 반응이 좋다" → 파일에 소비자 반응 데이터 없음 | `[데이터 없음 — 리뷰 내용 별도 수집 필요]` |
| 출처 혼용 | 파일A 수치를 파일B 출처로 표기 | 파일명과 컬럼명 재확인 후 수정 |
| 제조 정보 임의 기재 | "국내 ODM 제조사 활용" → 데이터에 없는 정보 | `[데이터 없음 — 고객사 확인 필요]` |
| 카테고리 혼용 제품명 | "마스크+바이브레이터 세트" → 서로 다른 카테고리 결합 | 단일 카테고리 제품만 제안. 검증 에이전트에서 불합격 처리 |
| 브랜드명 포함 | "코멧 스타일 마스크" → 경쟁사 브랜드명 포함 | 브랜드명 제거, 제품 특성으로 대체 |
| 자기합리화 | 불합리한 제안을 "데이터 기반"으로 정당화 | 검증 에이전트의 독립적 평가에 의존. 불합격 시 재생성 |
